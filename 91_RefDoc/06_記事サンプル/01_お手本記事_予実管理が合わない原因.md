# ▼ 構造原因（3つに分解）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 原因①：定義が揺れている

「売上」と言ったとき、全員が同じ数字を思い浮かべていますか？

- 税抜きなのか、税込みなのか
- 請求ベースなのか、検収ベースなのか
- 値引き・返品は控除済みなのか、別管理なのか

経理は「会計基準に沿った売上」を見ています。
事業部は「案件の着地見込み」を見ています。
経営企画は「予算との比較用に加工した売上」を見ています。

全員「売上」と言っているのに、数字が違う。
これが「定義が揺れている」状態です。

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：原因セクションの設計                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 見出しにキーワードを含める                                │
│    → 「定義が揺れている」（検索にヒットしやすい）            │
│                                                             │
│ ② 問いかけから始める                                        │
│    → 読者に「うちはどうだろう」と考えさせる                  │
│                                                             │
│ ③ 具体例を3つ並べる                                         │
│    → 箇条書きで読みやすく                                   │
│    → 「あ、うちもこれだ」と思わせる                         │
│                                                             │
│ ④ 部門ごとの視点の違いを示す                                │
│    → 「誰も悪くない、構造の問題だ」と納得させる              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 原因②：締めのタイミングがバラバラ

「今月の実績」はいつ確定しますか？

多くの会社で、こんな状態になっています：

- 経理は「月末締め、翌月10日確定」
- 事業部は「週次で速報を出している」
- 経営会議は「翌月5日に前月実績を報告」

経理の確定を待つと、経営会議に間に合わない。
だから事業部が「速報」を出す。
でも速報と確報で数字が違うから、「どっちが正しいの？」になる。

これが「締めがズレている」状態です。

---

## 原因③：参照元が宣言されていない

「正しい数字」はどこにありますか？

- 経理のExcel？
- 事業部のExcel？
- ERPの画面？
- BIダッシュボード？

「どこを見れば正しい数字がわかるか」が宣言されていないと、
各部門が「自分のExcel」を正として持ち始めます。

これが「参照元が散らばっている」状態です。

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：原因を3つに絞る理由                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 「3つ」が人間の認知に最適                                 │
│    → 2つだと物足りない、4つ以上だと覚えられない              │
│                                                             │
│ ② MECE感が出る                                              │
│    → 「この3つを押さえれば解決する」と思わせる               │
│                                                             │
│ ③ 各原因の構造を同じにする                                  │
│    → 問いかけ → 具体例 → 「これが〇〇の状態」で締める       │
│    → 読者が読み方を学習して、読みやすくなる                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

# ▼ 打ち手（90日ロードマップ）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 90日で「正の数字」を作るロードマップ

原因がわかったら、打ち手は明確です。
「定義」「締め」「参照元」を順番に整備します。

### Week 1〜2：指標定義書を作る（上位10指標だけ）

まず、経営会議で使う「上位10指標」だけ定義書を作ります。

| 指標名 | 計算式 | 粒度 | 例外処理 |
|-------|-------|-----|---------|
| 売上高 | 請求額−返品−値引 | 月次/部門別 | 前受は除外 |
| 営業利益 | 売上−売上原価−販管費 | 月次/全社 | 特別損益除外 |
| ... | ... | ... | ... |

**ポイント：全部やろうとしない。上位10指標で十分。**

---

### Week 3〜4：締めカレンダーを作る（二階建て運用）

「速報」と「確報」の二階建てで運用します。

```
┌──────────────────────────────────────────────────────┐
│ 締めカレンダー（例）                                  │
├──────────────────────────────────────────────────────┤
│ 月末    ：当月締め                                   │
│ 翌月3日 ：速報確定（経営会議用）                      │
│ 翌月10日：確報確定（経理確定）                        │
│ 確報後  ：速報→確報に差し替え                        │
└──────────────────────────────────────────────────────┘
```

**ポイント：速報は「暫定」と明記し、確報で必ず差し替える。**

---

### Month 2：参照元マップを作る（SoR宣言）

「この指標は、このシステムの、この画面を正とする」を宣言します。

| 指標名 | 参照元（SoR） | 備考 |
|-------|-------------|------|
| 売上高 | ERP > 売上管理 > 月次集計 | 経理確定後 |
| 受注残 | SFA > ダッシュボード | 毎週月曜更新 |
| 人件費 | 人事システム > 月次レポート | 翌月5日確定 |

**ポイント：「どこを見ればいいか」を1枚にまとめる。**

---

### Month 3：会議の問いを固定する

最後に、経営会議の「問い」を固定します。

```
経営会議の固定アジェンダ（例）
1. 前月実績サマリー（5分）
2. 予算比差異トップ3とその要因（10分）
3. 見通し修正の要否（10分）
4. 次月のアクション確認（5分）
```

**ポイント：「報告」ではなく「差異→打ち手」を問う形にする。**

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## よくある失敗パターン3選

### 失敗①：BIを入れたのにExcelが減らない

BIツールを導入しても、指標定義が曖昧なままだと、
「BIの数字が信じられない」と言われ、結局Excelで再計算される。

**→ BIの前に、指標定義書を作る。**

---

### 失敗②：締め短縮が「経理の根性」に依存

「もっと早く締めて」と言われ、経理が残業で対応。
担当者が異動したら、元に戻る。

**→ 速報/確報の二階建てにして、無理のない運用を設計する。**

---

### 失敗③：部門ごとのExcelが「正」になっている

参照元が宣言されていないと、各部門が「うちのExcelが正」と主張。
統合しようとすると「うちの運用が壊れる」と抵抗される。

**→ 参照元（SoR）を経営レベルで宣言し、例外を認めない。**

# ▼ 情シス/DXの方へ（固定コーナー）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 情シス/DXの方へ：ここが詰まりどころ

予実管理の整備は「システムを入れれば解決」ではありません。

指標の定義は、経営企画や経理が決めるものです。
締めのルールは、業務部門との合意が必要です。
参照元の宣言は、経営レベルの意思決定です。

情シス/DXの方にお願いしたいのは、
**「仕組みを作る前に、定義・締め・参照元の合意があるか確認すること」**です。

合意がないまま仕組みを作ると、
「システムの数字が信じられない」と言われ、Excelに戻ります。

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：情シス/DXコーナーの設計                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 毎回入れる「固定コーナー」                                │
│    → 主語（経営企画/FP&A）を崩さず、副読者を拾う            │
│                                                             │
│ ② 1段落で完結させる                                         │
│    → 長くしない。ここはサブ要素                             │
│                                                             │
│ ③ 記事ごとに1文だけ具体を変える                             │
│    → 今回は「定義・締め・参照元の合意があるか確認」         │
│    → 次回は別のポイントに差し替え                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 今日の持ち帰り：予実管理セルフチェック

あなたの会社の予実管理、この5つはできていますか？

- [ ] 経営会議で使う上位10指標の定義書がある
- [ ] 速報/確報の締め日と差し替えルールがある
- [ ] 「どこを見れば正しい数字がわかるか」が宣言されている
- [ ] 経営会議の問い（差異→打ち手）が固定されている
- [ ] 部門ごとの「勝手計算」を禁止/抑制する仕組みがある

**3つ以上チェックが付かなければ、まず指標定義書から始めてください。**

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：チェックリストの設計                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 5〜7項目に絞る                                            │
│    → 多すぎると使われない                                   │
│                                                             │
│ ② チェックボックス形式にする                                │
│    → 読者が「自分ごと」として使える                         │
│    → 社内共有されやすい（「これやってみて」）               │
│                                                             │
│ ③ 最後に「次のアクション」を添える                          │
│    → 「まず指標定義書から」と具体的に                       │
│                                                             │
│ ④ これが「保存」「スキ」を誘発する                          │
│    → Noteの伸びに直結                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**テンプレートが欲しい方へ：**
→ 指標定義書テンプレート【コピペで使える】（準備中）

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：次に読むリンクの設計                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
```
