# ▼ 構造原因（3つの設計が足りない）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 設計①：指標設計 —— 何を測るか、どう計算するか

KPIの第一の設計は「指標そのもの」です。
ここが曖昧だと、そもそも議論ができません。

### よくある問題

- 「顧客満足度」と言っても、NPSなのかCSATなのかアンケート回答率なのか不明
- 「売上成長率」の分母が前年同月なのか前月なのか人によって違う
- データソースが複数あり、どこから取った数字か分からない

### 指標設計で決めるべき4項目

| 項目 | 内容 | 例 |
|------|------|-----|
| 指標名 | 正式名称 | 月次受注額 |
| 計算式 | 分子/分母を明記 | 当月受注金額（税抜） |
| データソース | どこから取るか | SFA > 受注レポート |
| 更新頻度 | いつ更新されるか | 毎週月曜AM |

**ポイント：「指標辞書」として1枚にまとめる。**

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：指標設計セクションの意図                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 「当たり前」を言語化する                                   │
│    → 多くの会社が「指標名だけ」で運用している               │
│    → 計算式・データソースまで決めている会社は少ない         │
│                                                             │
│ ② テーブルで「何を決めればいいか」を明示                    │
│    → 読者がそのまま使える形式                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 設計②：責任設計 —— 誰がオーナーか、何の権限があるか

KPIの第二の設計は「責任」です。
ここが抜けると、数字を見ても誰も動きません。

### よくある問題

- KPIは経営企画が作ったが、現場は「押し付けられた」と思っている
- 「全社KPI」なので、誰の責任かが曖昧
- 数字が悪くても「それは他部門の影響」と言い訳できる

### 責任設計で決めるべき3項目

| 項目 | 内容 | 例 |
|------|------|-----|
| オーナー | この数字の責任者 | 営業本部長 |
| 権限範囲 | オーナーが動かせる範囲 | 営業人員配置、案件優先度 |
| 報告先 | 誰に報告するか | 経営会議（月次） |

**ポイント：オーナーには「動かせる権限」がなければ意味がない。**

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：責任設計の核心                                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 「オーナー」と「権限」をセットにする                       │
│    → 責任だけ押し付けて権限がないケースが多い               │
│    → これがKPI形骸化の最大の原因                            │
│                                                             │
│ ② 小泉さんの経験則を入れる余地                               │
│    → 「実際には、オーナーを決めても動かないケースがある」   │
│    → 「権限がないオーナーは、ただの報告係になる」           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 設計③：閾値設計 —— いつ動くか、何をするか

KPIの第三の設計は「閾値（しきいち）」です。
ここが抜けると、数字が悪くても「様子見」で終わります。

### よくある問題

- 目標値はあるが、「どこまで下がったらアクションするか」が決まっていない
- 「赤」になっても、会議で報告されるだけで何も起きない
- 改善アクションが属人的で、担当者によって対応が違う

### 閾値設計で決めるべき4項目

| 項目 | 内容 | 例 |
|------|------|-----|
| 目標値 | 達成すべき水準 | 月次受注額 1億円 |
| 警告閾値 | 注意喚起するライン | 8,000万円を下回ったら |
| 危険閾値 | 即時対応が必要なライン | 6,000万円を下回ったら |
| 定型アクション | 閾値を超えたら何をするか | 案件レビュー会議を招集 |

**ポイント：閾値を超えたら「自動的に」何かが起きる設計にする。**

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：閾値設計の核心                                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 「目標値」と「閾値」は違う                                 │
│    → 目標値：達成したい水準                                 │
│    → 閾値：アクションを起こすトリガー                       │
│                                                             │
│ ② 「定型アクション」まで決める                               │
│    → 「赤になったら考える」ではなく                         │
│    → 「赤になったら自動的に○○が起きる」                    │
│                                                             │
│ ③ これが「見るだけで終わらない」の核心                       │
│    → 閾値設計がないKPIは、ただのダッシュボード飾り          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

まず、既存のKPIを棚卸しして、**本当に意思決定に使うもの**だけに絞ります。

**絞り込みの基準：**
- 経営会議で議論されるか？
- この数字が悪いとき、誰かがアクションを取るか？
- データが定期的に更新されているか？

**ポイント：30個を7個に減らすだけで、議論の質が上がる。**

# ▼ アンチパターン（失敗しがちな落とし穴）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## KPI設計でよくある失敗パターン4選

### 失敗①：KPIを増やしすぎる

「測れるものは全部測ろう」と考えると、KPIが30個、50個と増殖します。
結果、どれも見られなくなり、「重要なKPI」が埋もれます。

**→ 経営会議で議論するKPIは7個以内に絞る。**

---

### 失敗②：「責任者不在」のKPIを作る

「全社売上」「顧客満足度」など、全社KPIを作ると、
誰の責任かが曖昧になり、数字が悪くても「他責」で終わります。

**→ 全社KPIは分解して、各部門のKPIに落とす。**

---

### 失敗③：閾値なしで「目標値だけ」設定する

「目標1億円」だけ設定して、途中経過をどう判断するか決めていない。
月末に「未達でした」と報告されても、もう手遅れ。

**→ 警告閾値（80%ライン）で早期アラートを出す。**

きれいなダッシュボードを作ると、「仕事した感」が出ます。
でも、誰も見ていない、見ても動かないダッシュボードは「飾り」です。

**→ ダッシュボードの前に、責任と閾値を設計する。**

## 情シス/DXの方へ：ここが詰まりどころ

KPIダッシュボードの構築を依頼されることがあると思います。

その際、ぜひ確認してほしいのは
**「このKPIのオーナーは誰で、閾値を超えたら何が起きるか」**です。

これが決まっていないままダッシュボードを作ると、
「きれいだけど誰も見ない」システムができあがります。

技術的には簡単に作れても、運用設計がなければ意味がありません。
**「責任と閾値の設計書をください」**と言える関係を作ってください。

```
┌─────────────────────────────────────────────────────────────┐
│ 💡 POINT：情シス/DXコーナーの設計                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ ① 今回の記事テーマに合わせた具体的な一言                     │
│    → 「責任と閾値の設計書をください」                       │
│                                                             │
│ ② 情シスの「あるある」に寄り添う                             │
│    → 「技術的には簡単に作れても」                           │
│    → 情シスの苦労を理解している姿勢                         │
│                                                             │
│ ③ 主語（経営企画）を崩さない                                 │
│    → 情シス向け記事ではなく、情シスを「拾う」だけ           │
│                                                             │

```

## 今日の持ち帰り：KPI設計セルフチェック

あなたの会社のKPI、この7つはできていますか？

**指標設計**
- [ ] 経営会議で使うKPIは7個以内に絞られている
- [ ] 各KPIの計算式とデータソースが明文化されている

**責任設計**
- [ ] 各KPIにオーナー（責任者）がいる
- [ ] オーナーには数字を動かす権限がある

**閾値設計**
- [ ] 目標値だけでなく、警告閾値・危険閾値が設定されている
- [ ] 閾値を超えたときの定型アクションが決まっている
- [ ] 閾値を超えたら「自動的に」何かが起きる仕組みがある

**4つ以上チェックが付かなければ、まずKPIの絞り込みから始めてください。**
