# 日本におけるNote記事生成・投稿自動化の先行事例調査

## 1. 先進的な実装事例ベスト5（技術的詳細）

1. **Playwrightを用いたnote記事自動投稿（Qiita）**
    
    - **技術スタック:** Node.js（Playwright）でブラウザを自動操作。
        
    - **自動化の範囲:** 記事タイトル・本文入力から**下書き保存**や**公開ボタンのクリック**までをフル自動化。環境変数で下書き保存モードと公開モードを切替可能。
        
    - **特筆すべき工夫:** Noteの**非公開API**が存在するものの利用は避け、ブラウザ操作で代替。サムネ画像のアップロード～トリミング確定、タグ入力まで自動で行うために細かな待機処理を実装。記事投稿後にスクリーンショットを保存する機能も備え、コードはChatGPTで生成後に人手でレビューした。
        
2. **OASIS – 複数プラットフォーム投稿自動化ツール（Zenn/GitHub）**
    
    - **技術スタック:** Pythonパッケージ「oasis-article」。各サービスのAPIやWeb操作を統合（Qiitaは公式API、Noteはブラウザ操作や内部API、WordPressはXML-RPC等）。Streamlit製のWeb UIも提供。LLM（GPT系）を組み込み、記事内容に基づくタグ推薦などを実施。
        
    - **自動化の範囲:** **Markdown原稿から**コマンド一発で**Qiita・Note・Zenn・WordPress**にクロス投稿。Noteについてはログイン情報を用いてブラウザ (Firefox) をヘッドレス起動し投稿処理。最新版ではNote内部APIへの直接HTML投稿に対応し高速化。
        
    - **特筆すべき工夫:** **一括クロス投稿**によりコピペの手間を削減。プラットフォーム毎の書式差異を吸収し（例: Markdown記法・埋め込みの調整、Mermaid図の画像レンダリング対応）、指定フォルダの画像をアイキャッチに自動設定する機能も搭載。さらに**カテゴリ・タグをLLMが自動提案**し、ユーザーは執筆に集中できる。
        
3. **LangChainエージェントによる記事ドラフト生成（Gluegent社ブログ）**
    
    - **技術スタック:** Python（LangChain）＋ OpenAI GPT-4 API。LangChainの**エージェント機能**でGoogle検索ツールを組み込み、必要に応じて外部情報を取得。生成記事は**GoogleドキュメントAPI**で自動出力し、権限設定まで行う。
        
    - **自動化の範囲:** ユーザーがテーマを入力すると、エージェントが**関連ニュースを検索→内容に基づき2000字超の記事を執筆→参考URLを記事末尾に自動挿入**するところまで実行（記事は下書きとしてDocs上保存）。公開は人間がブログに転載し調整。
        
    - **特筆すべき工夫:** **Retrieval Augmented Generation (RAG)** の実践例であり、**最新情報の取り込み**と**出典付き記事生成**を両立。LangChainエージェント（OpenAI Tools Agent）が検索クエリ発行から文章生成まで自律的に判断するため、執筆フローを自動化。将来的な発展として、**フリー画像API連携による画像挿入**や、過去ブログ記事を学習させて**章構成・文体を模倣**する計画も言及。
        
4. **Dify×n8nによる「バズるNote記事」分析・提案エージェント（note記事）**
    
    - **技術スタック:** ノーコードAI開発プラットフォーム**Dify**＋ワークフロー自動化**n8n**。Dify上でLLMエージェント（Anthropic Claude想定）を構築し、n8n経由でNoteの**非公式API**にアクセスおよびGoogleスプレッドシート連携。
        
    - **自動化の範囲:** ユーザーが興味あるテーマを入力すると、エージェントが**Note上で「スキ」（いいね）が多い記事**をキーワード検索し取得。人気記事のタイトル群を**AIで解析し共通要素を抽出**、その上で「バズる可能性の高い記事タイトル案」を根拠付きで複数生成。応用編では**人気記事リストを自動でスプレッドシート出力**する機能も追加。
        
    - **特筆すべき工夫:** Note内部で利用されている**検索API等のエンドポイントを逆利用**し、スクレイピングより安定して大量の記事メタデータを取得。得られたデータをAIが分析することで、**トレンド分析**＋**コンテンツ企画提案**という高度な支援を実現している。ノーコード主体で構成され、DifyのチャットフローDSLファイルが公開されており再利用も容易。
        
5. **n8n＋各種AIツールによるNote連携＆配信フロー（個人開発事例）**
    
    - **技術スタック:** オープンソースの自動化サーバー**n8n**上に独自ワークフローを構築。NoteのRSSフィードを監視して新規投稿をトリガーとし、HTTPリクエストノードやスクリプトでNoteに**ログインしてアクセストークンを取得**。LLM（OpenAI API）を組み合わせて記事要約とコメント文を生成。外部サービス（Discord・Misskey・Dev.to等）へのAPI投稿を実行。
        
    - **自動化の範囲:** Noteで記事を公開すると**即座にワークフロー起動**。公開記事の内容を取得・整形し、**Discordコミュニティへの投稿**、**記事全文をDifyナレッジベースに登録**（→Discord上のChatbotから過去記事を問い合わせ可能に）、**記事要約＋一言コメントの自動生成**、**Misskeyへの宣伝投稿**、**特定ハッシュタグが付いた記事の場合はDev.toへ英訳クロスポスト**…等、一連の拡散・活用処理を自動化。
        
    - **特筆すべき工夫:** **RSSフィード＋iPaaSによるイベント駆動**でNoteに公式APIが無い弱点を補完。Noteプレミアム会員向け記事も取得すべくログイン処理を組み込むなど高度なハック。さらに生成AIで自サイト専用の**チャットボット知識庫を構築**したり、記事内容を元に**宣伝文を自動生成**してマルチプラットフォームに投稿するなど、**オウンドメディア運用の省力化**を極限まで追求している。
        

## 2. 「品質担保」のためのアーキテクチャ・工夫

各事例のエンジニアは、AIによる誤情報や単調な文章にならないよう様々な工夫を凝らしています。

- **外部知識の活用（RAG）:** 信頼性向上のため、記事生成時に**検索エンジンや自社データベースから情報を取得**して引用する仕組みがあります。例えばGluegent社の例では、LangChainエージェントがGoogle検索で最新情報を収集し記事に反映しています。これによりAIが事実に反する内容をでっち上げるリスクを低減し、記事末尾に**参考URLを自動表示**してエビデンスを示しています。
    
- **マルチステップのプロンプト/エージェント設計:** **下書き→推敲→最終化**と段階を踏ませるアプローチです。Claude APIを使った事例では、**過去記事の分析(Subagent)**と**記事ドラフト生成(Subagent)**を分けて実装。まずAIが過去記事を解析し構成テンプレートや要点を抽出、次にその結果をもとに本文を執筆させています。段階的なワークフローにすることで、一度に全てを生成させるよりも**文脈や構成の整合性**が保ちやすく、誤りの検出・修正も容易になります。
    
- **文体・フォーマットのガイドライン提供:** 平坦な文章を避け自社らしいトーンを出すため、**スタイルガイド**をAIに埋め込んでいます。たとえばClaude Skillsを用いたケースでは、「句点は『。』で統一」「コロンの乱用禁止」といった**文体ルールをSkillとして定義**し、生成文が一定の書き味になるよう誘導しています。また自分の過去記事群をAIに読ませて**お手本**にすることで、用語の使い方や段落構成のパターンを学習させています。これによりAI生成文であっても従来の記事と乖離しない品質を担保しています。
    
- **人間によるレビュー・編集プロセス:** **Human in the Loop**を明確に設計に組み込むのも重要なポイントです。にもある通り、AIで下書きを生成しても「必ず最後は人の手で確認すること」が推奨されています。E.P.ラボの提言では、AIアシストでドラフト作成までは自動化しつつ、**公開前の最終チェックと投稿ボタン押下は人間が行う**という運用が現実的とされています。この人間の介在により、事実誤認の修正や文章表現の調整ができ、企業ブログとして信頼性の高い記事を発信できます。
    
- **その他のテクニック:** 長文生成時の暴走を防ぐため**適切なプロンプト長や温度パラメータの調整**、生成結果に対する別AIモデルでの**校正・事実チェック**、段落ごとに**ソースとの照合**を行うツールチェーンの併用など、品質向上の工夫が報告されています（※一部は一般論であり今回調査中の具体的事例には明示的な実装言及なし）。
    

## 3. Note連携の技術的ハック（TIPS）

公式API非提供のNoteと外部ツールを繋ぐために、エンジニア達は様々な**非公式手法**を試みています。

- **非公開APIの解析利用:** Noteウェブ版が内部で使用するAPIエンドポイントを調べ、直接HTTPリクエストを送る方法です。にあるように、検索用の`/v3/searches`や記事取得用の`/v3/notes/[記事ID]`などが判明しており、これらを叩くことでプログラムから記事データを取得できます。認証が必要な操作（自分の下書き一覧取得や投稿など）は、まず自動ログインして**トークン（セッションcookie）を取得**し、その情報をリクエストヘッダに付与して実行します。実装難易度は高いものの、**Note公式のUI変更に左右されにくく高速**に処理できるメリットがあります。逆に突然内部仕様が変更されるリスクや、利用規約上サポート対象外である点には注意が必要。
    
- **ブラウザ自動操作（スクレイピング）:** 最も汎用的な方法で、**SeleniumやPlaywright**を使いNoteの投稿画面を自動操作します。Qiita記事の例では「タイトル入力→本文入力→公開ボタンクリック」までヘッドレスブラウザで再現し、画面上のボタン要素やショートカットキー操作で投稿処理を行っています。メリットは人間と同じ操作であるため確実に動作する点ですが、動作がブラウザレンダリング待ちになるため**速度が遅く負荷も高め**です。またUIレイアウト変更の影響も受けやすいため、安定運用には随時メンテナンスが必要になります。
    
- **メール投稿や外部連携サービス:** （調査の範囲ではNoteにブログ記事投稿用のメール受信機能は確認できませんでした。そのため他サービス経由の投稿は一般的でないようです。ただしRSS出力は公式に提供されており、ZapierやMake経由で**投稿「後」処理を自動化する**用途には使われています。）
    
- **RSS＋iPaaSによる周辺業務自動化:** 上述の通りNoteは各ユーザーごとにRSSフィード（公開記事一覧XML）を提供しています。これを活用し、**Zapier/Make等のiPaaSで定期ポーリング→新着検知→SNS告知や社内Slack通知**といった連携が可能です。投稿そのものの自動化ではありませんが、「公開後の拡散」を自動化することでエンジニア自身の手間を減らしつつ公式機能の範囲内で行える安全な手法として推奨されています。
    
- **速度・精度向上のテクニック:** Noteへの投稿を大量に行う場合、ブラウザ操作の代わりに**内部APIを直接呼び出してHTMLコンテンツを投稿**することで大幅に処理を短縮できます。他にも、ブラウザ操作時に**ヘッドレスモード＋並列処理**で速度アップ、エラー時にスクリーンショットを保存してデバッグ容易にする工夫（Qiita例）など現場ならではのTipsが共有されています。
    

## 4. まとめ：EPM Note Engineへの採用推奨機能

今回調査した先行事例の知見を踏まえ、当社の「EPM Note Engine」（LangGraph + RAG + Streamlitベース）に組み込むと有用な機能・アイデアを以下に提案します。

- **エディタ支援とドラフト自動生成:** 執筆者がタイトルや箇条書き程度のアウトラインを入力すると、バックエンドで**LLMエージェントが関連情報を検索（社内ナレッジやWeb）**しつつ全文を書き起こす機能。Gluegent事例のように**参考文献URL付き**の下書きを生成し、執筆の初期負荷を大幅に下げます。社内ドキュメントをRAGのソースに含めれば、自社固有の専門知識を盛り込んだ記事ドラフトも可能です。
    
- **過去記事のスタイル学習と一貫性チェック:** 社内ブログの過去記事群をEmbeddingし、生成時にベクトル検索で類似記事の要約や口調を取り出してプロンプトに活用する仕組みです。Claude Skillによる文体ガイドライン適用やテンプレート生成のアプローチを取り入れ、**社内標準の語彙・トーンを踏襲**した文章を自動で作成します。また投稿前に文章校正ボットが内容をチェックし、過去記事との重複や不整合がないか検証するステップを組み込むことで品質を担保します。
    
- **Noteへのシームレス投稿（安全策含む）:** OASISやPlaywrightの知見を活かし、作成した記事を**ワンクリックでNoteに下書き保存・公開**できる機能を実装します。具体的には、Noteアカウント認証情報をもとに**ヘッドレスブラウザ操作**または**非公開API呼び出し**で記事を投稿するモジュールです。ただしNote運営ポリシー上グレーな部分もあるため、システム上で**「自動下書き保存まで」「完全自動公開まで」**などモード選択を可能にし、必要に応じて**最終確認は人間が行う**フローも選択できるようにします。これにより、自動化とコンプライアンスのバランスを取ります。
    
- **投稿後プロモーションの自動化:** 投稿が公開されたらその内容を元に**宣伝用のSNS投稿文や社内向け要約を自動生成**し、多チャネルに配信する機能です。具体例として、NoteのRSSフィードを監視して新規記事タイトルとURLを取得→AIが140字の告知文と適切なハッシュタグを生成→Twitter/XやLinkedInにAPI投稿、といった一連の処理をボタン一つで設定できます。の事例のようにMisskeyやDiscordへの展開、のように英訳して技術者コミュニティ（Dev.to等）へのクロスポストも可能でしょう。これらにUTMパラメータを付与してリンクを配信すれば、後日どれだけの流入があったか分析もできます。
    
- **ワークフローのカスタマイズ性とUI:** 最後に、上記の各工程を社内マーケターやブロガーが**ノーコードで組み替えられるUI**を用意すると理想的です。Streamlit製の管理画面から、「アイデア出し→下書き生成→編集→公開→拡散」のフローをオンオフ制御したり順序を入れ替えたりできると、利用部門のポリシーに柔軟に適応できます。例えば**「アイデア出しのみAI支援」**「**公開は手動**」「**公開後は自動で社内Slack通知**」といったカスタムがドラッグ&ドロップ操作で可能なイメージです。OASISがStreamlit UIを提供しているように、当エンジンも専門知識なしで扱えるフロントエンドを備えることで、より多くの現場に受け入れられるでしょう。
    

以上の機能を取り入れることで、単なる文章生成に留まらない包括的なNote記事作成エコシステム（仮称:EPM Note Engine）を実現できると考えます。先行事例に学びつつ、品質と効率の両立するシステムを目指しましょう。
